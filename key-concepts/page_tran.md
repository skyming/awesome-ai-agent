# awesome-ai-agent

 ## 论文翻译

 - **📊 [思维链提示激发大型语言模型的推理能力 (Chain-of-Thought Prompting Elicits Reasoning in Large Language Models)](https://arxiv.org/pdf/2201.11903)**
   
 摘要 (Abstract)
 本文探索了如何通过生成一系列中间推理步骤（我们称之为“思维链”，Chain-of-Thought）来显著提升大型语言模型进行复杂推理的能力。我们证明了，只要提供包含思维链的少样本提示（Few-shot prompting），足够大的语言模型就能展现出这种推理能力。我们在算术、常识和符号推理三大类基准测试上进行了实验。结果表明，思维链提示在超越特定规模的大模型上展现出了“涌现能力（Emergent property）”。例如，在 PaLM 540B 模型上使用仅包含 8 个示例的思维链提示，在 GSM8K 数学应用题基准测试中达到了当时的最先进（SOTA）水平，甚至超越了微调（fine-tuned）带有计算器的验证器模型。

 1. 引言 (Introduction)
 语言模型在很多自然语言任务上表现出色，但在涉及多步数学计算或逻辑推理的任务上却经常碰壁。传统的少样本提示（Standard Few-shot Prompting）往往无法很好地处理这些复杂任务。

 本文提出了一种简单而有效的方法：将“得出最终答案的中间思考过程”也加入到提示词中。这种方法的灵感来源于人类解决复杂问题时的认知过程——我们通常会把大问题拆解成几个小步骤来解决。我们将这种包含中间步骤的提示称为 Chain-of-Thought (CoT)。

 2. 思维链提示 (Chain-of-Thought Prompting)
 在这一节中，作者详细定义了什么是思维链。

 方法论： 如论文图1（即您之前发的图）所示，标准提示直接给出问题和答案；而思维链提示在给出答案前，会展示一段文字推导过程。

 四大优势：

 分解复杂性： 它允许模型将多步问题分解为中间步骤，这意味着可以将更多的计算资源分配给需要更多推理的问题。

 可解释性： 它为模型的行为提供了一个可解释的窗口，表明模型是如何得出特定答案的，也方便人类查错。

 通用性： 它不仅适用于数学题，还可以用于常识推理、符号操作等任何需要跨度思考的任务。

 易用性： 只需要极其少量的样本（比如 8 个示例），就能在大型现成模型上轻易激发这种能力，无需重新训练或微调模型。

 3. 算术推理实验 (Arithmetic Reasoning)
 作者在 5 个数学应用题数据集（GSM8K, SVAMP, ASDiv, AQuA, MAWPS）上测试了 5 个不同的大模型（涵盖参数量从 4 亿到 5400 亿不等）。

 核心发现 1：模型规模的“涌现”现象。 实验发现，思维链提示并不能改善小型模型（如小于 100B 参数）的表现，小模型即便生成了思维链，逻辑也是混乱的。只有当模型规模足够大（如 PaLM 540B, GPT-3 175B）时，思维链提示才会带来突破性的性能提升。这被称为一种“涌现能力”。

 核心发现 2：解决复杂问题更有效。 问题越复杂（需要更多步骤），思维链提示带来的性能提升就越显著。

 结果： PaLM 540B 结合思维链，在 GSM8K 测试集上准确率达到了 60.1%（当时的新纪录）。

 4. 常识推理实验 (Commonsense Reasoning)
 除了数学，作者还测试了需要常识和物理直觉的任务（如 CSQA, StrategyQA, Date Understanding, Sports Understanding）。

 现象： 对于 StrategyQA（需要组合多条世界知识来回答的策略问题）和 Date Understanding（日期推算），思维链提示同样大幅提高了模型的准确率。这证明了 CoT 不是仅限于数学计算的技巧，而是一种通用的逻辑强化手段。

 5. 符号推理实验 (Symbolic Reasoning)
 作者设计了两个测试：

 末字母拼接 (Last letter concatenation)： 比如问 "Elon Musk" 名字的最后字母拼接是什么（答案应该是 "nk"）。

 硬币翻转 (Coin flip)： 比如“硬币正面朝上。张三翻转了它，李四没翻，王五翻转了它。现在是正面还是反面？”

 结果： 面对这类需要过程追踪的符号任务，标准提示的表现极差（甚至低于随机猜测），而思维链提示让大模型以极高的准确率解决了这些问题，并且展现出了“长度泛化能力”（即提示中只展示 2 次翻转的例子，模型能够正确推理出 5 次翻转的结果）。

 6. 讨论与局限性 (Discussion and Limitations)
 作者在最后客观探讨了该方法的局限性：

 无法保证完全正确： 模型生成的“思维链”虽然有助于推理，但有时也会产生错误的中间步骤（幻觉），导致最终结论错误。

 推理代价： 在实际应用中，生成长序列的思维链需要消耗更多的计算资源（Token 生成成本更高）。

 对小模型不友好： 如前文所述，它依赖于百亿/千亿级别大模型本身的预训练底座，小模型无法使用这种方法。

 总结 (Conclusion)
 思维链（CoT）是一种极其简单却非常强大的机制。它证明了不需要改变模型架构或进行昂贵的微调训练，仅仅通过改变 Prompt（提供推理步骤的示范），就能释放大型语言模型深藏的推理潜力。

 - **📊 [思维树：使用大型语言模型进行深思熟虑的问题求解 (Tree of Thoughts: Deliberate Problem Solving with Large Language Models)](https://arxiv.org/abs/2305.10601)**

 摘要 (Abstract)
 大型语言模型（LLMs）在解决各类任务上表现出色，但它们目前的推理机制主要依赖于从左到右、标记级别（token-level）的顺序决策过程。这类似于人类直觉式的、“系统1”的思考方式，而在需要探索、战略性前瞻或在最初决策失败时能够回溯的复杂任务上表现不佳。
 为了克服这一限制，我们提出了一种新的语言模型推理框架——思维树（Tree of Thoughts, ToT）。它推广了经典的思维链（CoT）方法，通过探索多条推理路径来解决问题。ToT 将问题构建为对一棵“树”的搜索，树上的每个节点代表一个中间的“思维（Thought）”状态。通过结合语言模型的自我评估能力和经典搜索算法（如广度优先搜索 BFS、深度优先搜索 DFS），ToT 允许模型进行深思熟虑的探索、向前看（Lookahead）以及回溯（Backtracking）。
 我们在 24点游戏、创意写作和迷你填字游戏等需要复杂规划的任务上进行了测试。结果表明 ToT 取得了巨大的性能飞跃。例如，在 24 点游戏中，GPT-4 使用 CoT 只有 4% 的成功率，而使用 ToT 的成功率飙升至 74%。

 1. 引言 (Introduction)
 双过程理论（Dual Process Theory）认为人类有两种认知模式：
 - 系统 1： 快速、自动、无意识、直觉式（类似于当前 LLM 生成下个单词的方式）。
 - 系统 2： 缓慢、深思熟虑、有意识的规划（例如下棋时思考多步、数学解题时尝试不同公式）。

 目前的 LLM（即使使用了 CoT）本质上仍然被限制在“系统 1”的模式中——它们只能顺着一条路往下写，如果中间写错了一步，无法撤销重来。本文的动机就是赋予 LLM“系统 2”的能力。在 ToT 中，LLM 被用来生成多种可能的中间步骤（节点），并主动评估这些步骤的质量，最后利用经典的搜索算法寻找全局最优解。

 2. 背景与现有方法对比 (Background)
 论文对比了目前主流的几种 Prompting 方法：
 - Input-Output (IO) Prompting： 标准提示，直接问问题，直接给答案。
 - Chain-of-Thought (CoT)： 思维链，要求模型输出一条单一的中间推理路径。
 - Self-Consistency with CoT (CoT-SC)： 思维链+自我一致性。让模型生成多条不同的思维链，最后通过投票选出出现次数最多的答案。这虽然有多条路径，但路径之间是不互动的，且都是盲目生成到底。
 - Tree of Thoughts (ToT)： 本文方法。维护一棵树，在每一个中间节点，模型会生成多个分支选择，对其进行评估，然后有策略地选择最有希望的分支继续探索。

 3. 思维树框架核心组件 (Tree of Thoughts Framework)
 构建一个 ToT 求解器，需要定义以下四个核心部分：
 - 思维分解 (Thought Decomposition)： 将复杂问题拆解为哪些中间步骤？例如，在数学题中，一个“思维”可能是一个方程；在写作中，一个“思维”可能是一段大纲。
 - 思维生成器 (Thought Generator)： 如何根据当前状态生成下一步的候选“思维”？
   - Sample（采样）： 如果思维空间很大（比如写作），就让模型使用不同的温度参数生成多个不同的下一段。
   - Propose（提议）： 如果思维空间受限（比如 24点游戏的下一步计算），就用一条 Prompt 让模型一次性列出所有可能的下一步。
 - 状态评估器 (State Evaluator)： 这是 ToT 最核心的创新。 如何评估当前分支是否值得继续走下去？论文利用 LLM 自身作为裁判：
   - Value（打分）： 让模型给当前状态打分（例如：一定能成功、可能成功、不可能成功）。
   - Vote（投票）： 当有多个不同路径时，让模型对比它们，投票选出最好的一条。
 - 搜索算法 (Search Algorithm)： 结合传统的启发式搜索算法来遍历这棵树：
   - 广度优先搜索 (BFS)： 比如 24点游戏，每一步保留最好的几个分支，抛弃差的，齐头并进。
   - 深度优先搜索 (DFS)： 比如填字游戏，先顺着一个方向填，如果发现填不下去了（死胡同），就回溯 (Backtrack) 到上一个状态换个词填。

 4. 实验证明 (Experiments)
 作者在三个极其考验规划和试错能力的任务上进行了测试：
 - 任务 1：24点游戏 (Game of 24)
   - 规则： 给 4 个数字，通过加减乘除算出 24。
   - 结果： 这是一个极容易走入死胡同的游戏。传统 CoT 让模型直接算，成功率只有 4%。ToT 使用 BFS（每一步尝试所有组合，评估出最有希望的留下），成功率达到了 74%。
 - 任务 2：创意写作 (Creative Writing)
   - 规则： 给出 4 个毫不相干的随机句子，要求写一篇文章，包含 4 个段落，且这 4 个段落必须分别以这 4 个句子结尾。
   - 结果： ToT 采用了先写多个大纲 -> 评估选出最佳大纲 -> 再写正文的步骤。无论是机器评估还是人类盲测，ToT 生成的文章连贯性都远超标准提示。
 - 任务 3：迷你填字游戏 (Mini Crosswords)
   - 规则： 完成一个 5x5 的字谜填空。
   - 结果： 这是一个强依赖“回溯”的任务（填错一个字母，后面的词就废了）。ToT 使用了 DFS + 回溯机制，成功率远超 CoT（从不足 20% 提升到了 60% 以上）。

 5. 讨论与局限性 (Discussion & Limitations)
 - 优点： ToT 范式将传统计算机科学中的“启发式搜索”与大模型的“生成/评估能力”完美结合，解锁了解决极高难度问题的能力。
 - 局限性 (代价)： ToT 的计算成本非常高昂。为了解决一个问题，它可能需要调用几十次大模型 API 来生成分支和进行评估，消耗大量的 Token。
 - 结论： 作者指出，对于简单的日常任务，不需要用 ToT（标准提示或 CoT 就够了）。ToT 主要是为需要复杂逻辑、长线规划、需要试错和系统 2 思考的高难度场景准备的（例如复杂代码编写、科研推理、长篇小说创作等）。

 总结：
 如果说 CoT 是一条道走到黑，ToT 就是在脑海中画出了一棵决策树——“如果我走A路线，下一步会怎样？如果走B路线，下一步又会怎样？A路线好像走不通了，我退回上一步选C路线吧。” 这让 AI 真正具备了类似人类的“深思熟虑”能力。

 - **📊 [具有语言强化学习能力的语言智能体 (Reflexion: Language Agents with Verbal Reinforcement Learning)](https://arxiv.org/abs/2303.11366)**

 摘要 (Abstract)
 大型语言模型（LLMs）越来越多地被用作与外部环境（例如游戏、编译器、API）交互的目标驱动型智能体。然而，传统的强化学习（RL）需要大量的训练数据和昂贵的模型权重微调。
 本文提出了一种全新的框架——Reflexion（反思）。该框架不是通过更新模型权重来强化语言智能体，而是通过语言反馈（Linguistic Feedback）来实现。具体而言，Reflexion 智能体会用自然语言对任务的反馈信号进行“口头”反思，然后将这些反思文本保存在一个情节记忆缓冲区（Episodic Memory Buffer）中，以便在后续的尝试中做出更好的决策。
 Reflexion 具有极高的灵活性，可以整合各种类型（如标量分数或自由格式语言）和来源（外部环境或内部模拟）的反馈信号。实验表明，Reflexion 在多项不同类型的任务（顺序决策、代码生成、语言推理）中，都显著超越了基线智能体的表现。

 1. 引言 (Introduction)
 人类在面对新任务失败时，往往会停下来思考：“我刚才哪里做错了？下次我该怎么做才能避免这个错误？”——这就是自我反思（Self-reflection）的能力。
 当前的 LLM 智能体（如基于 ReAct 框架的智能体）虽然能推理和行动，但它们往往会在同一个地方反复跌倒。如果让它们通过传统的强化学习去纠错，不仅成本极其高昂，而且对于闭源模型（如 GPT-4）来说根本不可行。
 因此，作者提出了 Reflexion：将传统的“标量奖励信号”（比如游戏里扣了 1 分）转化为“丰富的语言反馈信号”（比如一段文字：“你刚才往左走撞到了墙，因为你没有提前查看地图，下次你应该先使用‘查看地图’的动作”）。智能体将这些经验作为额外的上下文（Context）带入下一次尝试中。

 2. Reflexion 框架的核心组件 (The Reflexion Framework)
 Reflexion 框架包含三个关键的协同模型：
 - Actor（行动者）： 负责根据当前的状态和目标，生成实际的文本和动作（Action）。论文中通常使用基于 CoT 或 ReAct 提示的大语言模型来充当 Actor。
 - Evaluator（评估者）： 负责评估 Actor 刚才生成的轨迹（Trajectory）的表现，并给出一个评价信号。
   - 在有些任务中，评估者是外部系统（比如代码编译器，如果代码跑不通它会报错）。
   - 在有些任务中，评估者是由大模型自己扮演的，它会根据特定启发式规则（比如判断智能体是不是陷入了无意义的重复循环）来判定当前尝试是否失败。
 - Self-Reflection 模型 (自我反思模型)： 这是整个框架的灵魂所在。 当 Evaluator 判定当前尝试失败时，Self-Reflection 模型会介入。它会阅读刚才失败的完整交互历史，并生成一段具体的“反思笔记”。
 - Memory (记忆缓冲区)： 智能体将上述生成的“反思笔记”存入记忆中。在开启下一轮全新尝试时，系统提示词中会附加一句话：“你之前尝试过这个任务但失败了，以下是你自己总结的经验教训：[反思笔记]”。

 3. 实验证明 (Experiments)
 作者在三种不同类型的核心任务上验证了 Reflexion 的威力：
 - 任务 1：顺序决策任务 (Sequential Decision-Making - AlfWorld)
   - 场景： 这是一个基于文本的家庭环境模拟器任务（比如要求智能体“找到一个苹果并把它放在桌子上”，智能体需要在各种房间里穿梭、翻箱倒柜）。
   - 结果： 基线 ReAct 智能体的成功率很快就停滞不前，而加入了 Reflexion 后，成功率从约 70% 飙升到了 97%。
 - 任务 2：知识推理问答 (Reasoning - HotpotQA)
   - 场景： 这是一个需要利用外部维基百科搜索引擎进行多跳推理的复杂问答测试。
   - 结果： 当智能体搜索方向错误导致找不到答案时，反思机制会教导智能体：“刚才搜索的关键词太具体了导致没有结果，下次应该尝试搜索更宏观的实体”。
 - 任务 3：代码生成 (Coding - HumanEval / MBPP)
   - 场景： 让大模型写一段 Python 代码，并在后台实际运行单元测试（Unit Tests）。
   - 结果： 如果代码报错，Reflexion 会让大模型阅读错误日志（Traceback）和失败的测试用例，并用自然语言解释哪里写错了。GPT-4 在 HumanEval 上的准确率从 67% 提升到了 88%。

 4. 讨论与优势 (Discussion & Advantages)
 - 轻量且免微调： 不需要改变语言模型内部的任何参数，直接利用了现有大模型强大的上下文学习能力。
 - 更细致的反馈方向： 传统的强化学习只告诉模型“你得分为 0”，而 Reflexion 用自然语言直接指出了“你错在哪里，应该向什么方向改进”，这是一种极其高效的“语义梯度（Semantic Gradient）”。
 - 可解释性强： 程序员可以明确看到智能体在记忆缓冲区里写下的反思日记。

 局限性 (Limitations)
 - 上下文窗口限制： 智能体不能无限次地反思和积累经验，因为反思笔记越来越长，最终会撑爆大模型的 Context Window。
 - 过度自信与反思幻觉： 大模型有时候会对自己错误的行为强行给出看似合理实则荒谬的“反思”。

 总结：
 《Reflexion》这篇论文提出了一个非常符合直觉的人类行为模仿范式：尝试 -> 失败 -> 评估反馈 -> 语言总结反思 -> 带着记忆重新尝试。

 - **📊 [在语言模型中协同推理与行动 (ReAct: Synergizing Reasoning and Acting in Language Models)](https://arxiv.org/abs/2210.03629)**

 摘要 (Abstract)
 尽管大型语言模型（LLMs）在复杂推理（如思维链 CoT）和在交互式环境中的行动（Action-only）方面分别取得了令人瞩目的进展，但这两种能力通常是分开研究的。
 本文提出了 ReAct 框架，旨在将大模型的推理（Reasoning）和行动（Acting）交织在一起。通过这种协同：
 - 推理帮助行动： 模型能够诱导、跟踪和更新行动计划，甚至处理交互中出现的异常情况。
 - 行动帮助推理： 模型能够与外部知识库或环境（如维基百科、API）接口，收集额外的信息以支持推理，从而有效缓解模型由于内部知识盲区导致的“幻觉（Hallucination）”。
 我们在问答、事实验证、基于文本的游戏和网页导航等多种任务上验证了 ReAct，证明了它在提高准确率、可解释性和可信度方面的巨大优势。

 1. 引言 (Introduction)
 人类智能的一个独特标志是，我们能够将“任务导向的行动”与“口头推理”无缝结合。比如你在厨房做饭时，会经历这样的循环：
 - 推理： “菜谱上说需要切洋葱，我得找把刀。”
 - 行动： （拉开抽屉寻找刀具）
 - 观察： （发现抽屉里没有刀）
 - 推理更新： “不在抽屉里，可能在洗碗机里，我去看看。”

 在 ReAct 提出之前，大模型的应用存在两条割裂的路线：
 - 纯推理 (Reasoning-only, 如 CoT)： 模型极易产生事实性错误（幻觉）。
 - 纯行动 (Acting-only)： 模型缺乏大局观和过程状态的维持，稍微遇到困难就会陷入死循环。
 ReAct 的核心思想就是将两者结合：让模型在执行下一个动作前，先“大声说出”自己的思考过程。

 2. ReAct 框架的核心范式 (The ReAct Framework)
 ReAct 将 Agent 的运行过程标准化为一个严格的交替循环机制，通常被称为 Thought-Action-Observation（思考-行动-观察） 循环：
 - Thought (思考)： 模型对当前的状态进行推理分析。它决定了当前的阶段性目标是什么，需要什么信息。
 - Action (行动)： 根据思考的结果，模型从预定义的工具库中选择一个具体的工具（函数/API）并传入参数。
 - Observation (观察)： 外部环境执行完动作后，将结果返回给模型。
 模型将这三个步骤的文本拼接在上下文中，继续生成下一个 Thought，不断循环，直到得出最终答案 Finish(答案)。

 3. 实验证明与应用场景 (Experiments & Scenarios)
 论文在两大类场景中验证了 ReAct 的有效性：
 - 场景一：知识密集型推理任务 (Knowledge-Intensive Reasoning - HotpotQA, FEVER)
   - 痛点： CoT 在遇到不知道的知识时会凭空捏造。
   - ReAct 的表现： ReAct 通过 Search 动作去维基百科查资料，然后再基于真实资料进行推理。这几乎根除了这部分推理的幻觉问题。
 - 场景二：具身/决策任务 (Decision Making - ALFWorld, WebShop)
   - 痛点： 纯行动模型由于没有工作记忆，经常在两个页面之间反复横跳。
   - ReAct 的表现： 引入 Thought 后，模型会在行动前写下状态日志。这种带有状态追踪的推理，使任务成功率分别提升了 34% 和 10%。

 4. 讨论与深远影响 (Discussion & Impact)
 - 极高的可解释性与纠错性： 开发者可以通过查看 Agent 输出的 Thought 日志，清晰地知道它在想什么。
 - 极简的少样本提示 (Few-shot setup)： 仅仅需要给模型看几个 Thought -> Action -> Observation 的文本样例，大模型就能瞬间领悟这种工作模式。

 核心总结与工程意义：
 ReAct 实际上不仅仅是一篇学术论文，它直接定义了现代 AI Agent 的工程标准。这个闭环，正是 ReAct 赋予整个行业的标准协议。

 - **📊 [语言模型可以自学使用工具 (Toolformer: Language Models Can Teach Themselves to Use Tools)](https://arxiv.org/abs/2302.04761)**

 摘要 (Abstract)
 现有的 LLM 存在一些短板：它们会在基础的算术上出错，会产生事实性幻觉，或者不知道当前的日期。而这些问题，对于一个极其简单的计算器或搜索引擎代码来说根本不是问题。
 Toolformer 提出了一种让语言模型通过自监督（Self-supervised）的方式，自主决定何时调用 API、传递什么参数，以及如何将返回结果无缝融入到后续文本生成中的方法。

 1. 引言 (Introduction)
 如果能让模型使用工具，就能完美弥补其缺陷。Toolformer 的核心突破在于：让大模型利用上下文学习能力来“刷题”生成自己的微调数据，从而学会使用工具。

 2. 核心方法：模型如何“自学”？ (The Approach)
 Toolformer 的训练过程分为三个关键步骤：
 - 采样 API 调用 (Sample)： 给模型一段普通的文本，让它尝试在任意觉得合适的地方，插入各种 API 调用指令。
 - 执行并过滤 (Execute & Filter)： 模拟执行这些 API，拿到真实的返回结果。系统会进行对比测试：加入这个 API 的真实结果后，模型预测原文本下一个词的交叉熵损失（Loss）是否下降了？如果 Loss 显著下降，则保留该样本。
 - 微调 (Finetune)： 将过滤后保留下来的十万级“成功调用样例”作为训练数据，对大模型进行微调。

 3. 实验配置与工具集 (Tools)
 Meta 的研究团队为 Toolformer 接入了 5 个基础工具 API：
 - 计算器 (Calculator)： 解决算术薄弱问题。
 - 问答系统 (Q&A System)： 解决特定实体的精确知识检索。
 - 搜索引擎 (Search Engine)： 解决长尾知识和时效性问题。
 - 翻译系统 (Machine Translation)： 解决低资源语种的翻译短板。
 - 日历 (Calendar)： 让模型拥有获取当前时间戳的概念。

 4. 实验结果与意义 (Results & Significance)
 - 降维打击大模型： 基于 6.7B 参数的 GPT-J 训练出的 Toolformer，在多项零样本测试中的表现，暴打了原本比它大几十倍的基础模型（如 175B 的 GPT-3）。
 - 保持核心能力： 学会使用工具并没有损害模型原有的语言连贯性和文本生成能力。
 - 范式转移： Toolformer 证明了，大模型完全有能力通过自监督微调，变成一个熟练识别时机并操作各类外部 API 的“中控大脑”。

 总结：
 Toolformer 开启了大模型从“纯文本生成器”向“工具调用中控”转变的新范式。
